# 附錄 B — 教學指南

> 給教師的 18 週課程規劃與評量設計。

---

## 課程定位

| 項目 | 說明 |
|------|------|
| 課程名稱 | AI Agent 原理與治理實務 |
| 適合對象 | 資訊科系大三以上 / 研究所 / 在職進修 |
| 前置課程 | 程式設計、版本控制基礎 |
| 每週時數 | 3 小時（1 小時講授 + 2 小時實作） |
| 評量方式 | 作業 40% + 期中報告 20% + 期末專案 40% |

## 18 週課程規劃

### Part 1：基礎篇（Week 1-5）

| 週次 | 主題 | 章節 | 實作 |
|------|------|------|------|
| W01 | 課程導論 + AI Agent 概觀 | Ch01 | 安裝 Claude Code，探索一個開源專案 |
| W02 | Agent 架構深入 | Ch02 | 畫架構圖，估算 Token，觀察工具使用 |
| W03 | 第一個 Agent 互動 | Ch03 | 用 Agent 完成小型專案初始化 |
| W04 | 提示工程基礎 | Ch04 | 提示改寫練習，A/B 比較實驗 |
| W05 | 提示工程進階 + 複習 | Ch04 | 設計 CLAUDE.md，角色扮演實驗 |

### Part 2：實戰篇（Week 6-8）

| 週次 | 主題 | 章節 | 實作 |
|------|------|------|------|
| W06 | SDD-BDD-TDD 工作流 | Ch05 | 用四問題法評估功能，寫 BDD 情境 |
| W07 | 工具系統與 MCP | Ch06 | MCP 伺服器實驗，比較有無 Context7 |
| W08 | 實戰整合 + **期中報告** | Ch05-06 | 用完整工作流開發一個小功能 |

### Part 3：治理篇（Week 9-13）

| 週次 | 主題 | 章節 | 實作 |
|------|------|------|------|
| W09 | 治理憲法 | Ch07 | 撰寫專案的 constitution.md |
| W10 | 紅線與護欄 | Ch08 | 設計語意禁止規則和審查機制 |
| W11 | 記憶與決策 | Ch09 | 建立記憶系統，記錄技術決策 |
| W12 | 品質閘門 | Ch10 | 建立自動化品質管線 |
| W13 | 治理整合 | Ch07-10 | 整合所有治理元件到一個專案 |

### Part 4：進階篇（Week 14-18）

| 週次 | 主題 | 章節 | 實作 |
|------|------|------|------|
| W14 | 多 Agent 協作 | Ch11 | 用多 Agent 做程式碼審計 |
| W15 | 生產部署與成本控制 | Ch12 | 設計團隊 Agent 使用規範 |
| W16 | 畢業專案工作坊 | Ch13 | 開始建造完整治理框架 |
| W17 | 畢業專案完成 | Ch13 | 完成框架，互相測試 |
| W18 | **期末發表** | 全部 | 展示治理框架，同儕評審 |

## 評量設計

### 作業（40%）

| 作業 | 週次 | 內容 | 配分 |
|------|------|------|------|
| HW1 | W02 | Agent 架構分析報告 | 5% |
| HW2 | W04 | 提示工程 A/B 實驗報告 | 5% |
| HW3 | W06 | 用 SDD-BDD-TDD 開發一個功能 | 10% |
| HW4 | W09 | 撰寫專案治理憲法 | 5% |
| HW5 | W10 | 設計語意禁止與審查規則 | 5% |
| HW6 | W12 | 建立品質閘門管線 | 5% |
| HW7 | W14 | 多 Agent 協作實驗 | 5% |

### 期中報告（20%）

```
主題：Agent 驅動開發實戰報告

要求：
1. 使用 SDD-BDD-TDD 工作流開發一個小功能
2. 記錄完整的開發過程（含 Agent 對話摘要）
3. 分析 Agent 的貢獻和局限
4. 評估 Token 使用效率

評分標準：
- 工作流完整度：30%
- 分析深度：30%
- 反思品質：20%
- 報告品質：20%
```

### 期末專案（40%）

```
主題：打造完整的 Agent 治理框架

要求：
1. 完成 Ch13 畢業專案的所有步驟
2. 在真實專案中驗證框架的有效性
3. 撰寫治理框架的使用指南
4. 同儕互相測試（用對方的框架做任務）

評分標準：
- 框架完整度：25%
  □ constitution.md
  □ semantic-deny.md
  □ human-review.md
  □ 記憶系統
  □ 品質管線

- 實用性：25%
  □ 能在真實專案中使用
  □ 規則具體可執行
  □ 不會過度限制 Agent

- 驗證結果：25%
  □ 5 個測試場景通過
  □ 同儕測試回饋正面
  □ Token 使用合理

- 文件品質：25%
  □ 使用指南清晰
  □ 規則有理由說明
  □ 有演化和維護計畫
```

## 教學建議

### 授課方式

```
每堂課的結構（3 小時）：

前 15 分鐘：回顧上週作業，討論遇到的問題
    │
    ▼
45 分鐘：概念講授（用真實案例和示範）
    │
    ▼
15 分鐘：休息
    │
    ▼
90 分鐘：實作練習（學生動手，老師巡場指導）
    │
    ▼
最後 15 分鐘：總結和下週預告
```

### 學生分組

```
建議 2-3 人一組：
- 共同使用一個 Agent 治理框架
- 輪流擔任「人類審查者」角色
- 互相測試對方寫的規則
- 共同完成期末專案
```

### 常見教學挑戰

| 挑戰 | 建議 |
|------|------|
| 學生沒有 API 額度 | 使用學校帳號或申請教育優惠 |
| 學生過度依賴 Agent | 要求寫「我學到了什麼」反思日誌 |
| 治理規則太抽象 | 用真實的 Bug 案例來說明規則的價值 |
| 學生程度差異大 | Lite/Standard/Full 三級制度自然分流 |
| 技術更新太快 | 強調原則而非具體工具，原則是持久的 |
