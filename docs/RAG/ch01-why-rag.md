## 開卷考試 vs 閉卷考試

讓我們用一個你一定熟悉的比喻來理解 RAG。

### 閉卷考試（傳統 LLM）

```
情境：高中期末考

學生：LLM（GPT-4、Claude 等）
考試方式：閉卷，只能靠記憶回答

特點：
  ✅ 記住了大量通用知識
  ✅ 反應超快，不用查書
  ❌ 只知道訓練截止日前的資訊
  ❌ 面對沒讀過的內容會亂猜
  ❌ 無法更新知識（不能偷讀小抄）
```

### 開卷考試（RAG 系統）

```
情境：研究所口試

學生：RAG 系統
考試方式：可以帶參考資料，邊查邊答

特點：
  ✅ 可以查詢最新資訊
  ✅ 答案有文件依據
  ✅ 知識可以隨時更新
  ✅ 能夠查閱私有資料
  ❌ 需要先建立資料庫
  ❌ 查資料需要額外時間
```

**RAG 就是讓你的 AI 從「閉卷考試」變成「開卷考試」。**

## AI 的記憶分類學

在深入 RAG 之前，我們要先搞清楚 AI 有哪些「記憶」。

```
AI 的記憶類型
════════════════════════════════════════════════════

  參數記憶（Parametric Memory）
  ┌─────────────────────────────────┐
  │  儲存在模型權重裡               │
  │  就像人類的長期記憶             │
  │  訓練完成後就固定了             │
  │  例：GPT-4 知道「台灣在亞洲」   │
  └─────────────────────────────────┘
            ↓ 問題
  ┌─────────────────────────────────┐
  │  ❌ 知識有截止日                │
  │  ❌ 無法存入私有資訊            │
  │  ❌ 更新成本極高（重新訓練）    │
  └─────────────────────────────────┘

  非參數記憶（Non-Parametric Memory）
  ┌─────────────────────────────────┐
  │  儲存在外部資料庫裡             │
  │  就像人類的參考書               │
  │  可以隨時新增和更新             │
  │  例：你公司的產品規格文件       │
  └─────────────────────────────────┘
            ↓ RAG 的作用
  ┌─────────────────────────────────┐
  │  ✅ 知識隨時可更新              │
  │  ✅ 支援私有和機密資料          │
  │  ✅ 不需要重新訓練模型          │
  └─────────────────────────────────┘
```

## 幻覺（Hallucination）：AI 的最大敵人

在介紹解決方案之前，我們要先了解問題的嚴重程度。

**幻覺**是指 AI 生成了聽起來合理但實際上是假的資訊。

### 真實案例

```
用戶問：「我們的 API 版本 3.2 支援哪些認證方式？」

❌ 沒有 RAG 的 AI 回答：
「API v3.2 支援 OAuth 2.0、API Key、JWT Token 和
 Basic Authentication 四種認證方式。您可以在設定頁面
 的 Security 選項中進行配置。」

   → 聽起來非常專業，但全部是捏造的
   → 你的文件根本沒有 v3.2，只有 v3.1 和 v4.0

✅ 有 RAG 的 AI 回答：
「根據您提供的 API 文件，我查詢了版本資訊：
 目前可用的版本為 v3.1 和 v4.0，沒有 v3.2 版本。

 v3.1 支援：OAuth 2.0、API Key
 v4.0 支援：OAuth 2.0、API Key、JWT Token

 您需要的是哪個版本的資訊？」
```

差距就是這麼大。

## RAG 的三步驟架構

RAG 的核心邏輯非常直覺，就三個字：**查、加、答**。

```
RAG Pipeline（管線）
════════════════════════════════════════════════════════════

  用戶問題
     │
     ▼
┌─────────────────────────────────────────────────────────┐
│  Step 1: RETRIEVE（檢索）                               │
│                                                         │
│  「這個問題跟哪些文件有關？」                           │
│                                                         │
│  用戶問題 → 向量搜尋 → 最相關的文件片段                │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  Step 2: AUGMENT（增強）                                │
│                                                         │
│  「把相關文件加進提示詞裡」                             │
│                                                         │
│  原始問題 + 相關文件 → 增強後的提示詞                  │
└────────────────────────┬────────────────────────────────┘
                         │
                         ▼
┌─────────────────────────────────────────────────────────┐
│  Step 3: GENERATE（生成）                               │
│                                                         │
│  「根據找到的資料回答問題」                             │
│                                                         │
│  增強後的提示詞 → LLM → 有依據的答案                   │
└─────────────────────────────────────────────────────────┘
```

## Fine-tuning vs RAG：你到底需要哪個？

這是初學者最常問的問題。讓我們用一個表格來比較：

| 比較項目 | Fine-tuning | RAG |
|----------|-------------|-----|
| **訓練成本** | 非常高（幾萬到幾十萬美金） | 低（建索引就好） |
| **更新知識** | 需要重新訓練 | 只需更新資料庫 |
| **私有資料** | 可以，但有資安風險 | 可以，資料在你手上 |
| **知識時效性** | 固定在訓練時 | 隨時可更新 |
| **適合場景** | 改變模型「說話方式」 | 增加模型「知道什麼」 |
| **可解釋性** | 黑箱，難以追蹤 | 可以看到參考來源 |

### 何時用 Fine-tuning？

```
✅ 你想讓 AI 用特定的語氣說話（例：像客服一樣親切）
✅ 你有特殊格式的輸出需求（例：永遠回傳 JSON）
✅ 你需要模型學習特定領域的語言習慣
```

### 何時用 RAG？

```
✅ 你有大量需要查閱的文件（FAQ、產品規格、合約）
✅ 你的知識需要頻繁更新
✅ 你需要答案有文件依據（可以引用來源）
✅ 你的預算有限（RAG 便宜很多）
```

> 💡 **實際上，很多生產系統會同時使用兩者**：用 Fine-tuning 調整模型的風格和格式，用 RAG 提供最新的知識。

### 沒有 RAG 的版本

```python
from openai import OpenAI

client = OpenAI()

def ask_without_rag(question: str) -> str:
    """
    直接問 LLM，不提供任何外部資料。
    這模擬了「閉卷考試」的情況。
    """
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": "你是一個客服助理，請回答用戶的問題。"
            },
            {
                "role": "user",
                "content": question
            }
        ]
    )
    return response.choices[0].message.content

# 測試
question = "你們的退款政策是什麼？週末可以退款嗎？"
print("=== 沒有 RAG 的回答 ===")
print(ask_without_rag(question))
print()
```

### 有 RAG 的版本（簡易版）

```python
def simple_retrieve(question: str, documents: str) -> str:
    """
    超簡單的「檢索」：直接把整份文件塞進去。
    （真實的 RAG 會更聰明地只選相關片段）
    """
    # 步驟 1：準備增強後的提示詞
    augmented_prompt = f"""
以下是公司的 FAQ 文件：

{documents}

---

根據上面的 FAQ 文件，請回答用戶的問題。
如果文件中沒有相關資訊，請明確說「文件中沒有此資訊」，不要猜測。

用戶問題：{question}
"""
    return augmented_prompt


def ask_with_rag(question: str, documents: str) -> str:
    """
    用 RAG 回答問題：先檢索相關文件，再生成答案。
    這模擬了「開卷考試」的情況。
    """
    # 步驟 1 & 2：檢索 + 增強
    augmented_prompt = simple_retrieve(question, documents)

    # 步驟 3：生成
    response = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=[
            {
                "role": "system",
                "content": "你是一個誠實的客服助理。只根據提供的文件回答問題。"
            },
            {
                "role": "user",
                "content": augmented_prompt
            }
        ]
    )
    return response.choices[0].message.content


# 測試
question = "你們的退款政策是什麼？週末可以退款嗎？"
print("=== 有 RAG 的回答 ===")
print(ask_with_rag(question, company_faq))
```

### 對比測試：提問一個文件沒有的問題

```python
def compare_responses(question: str, documents: str):
    """
    並排比較有無 RAG 的差異。
    """
    print(f"問題：{question}")
    print("=" * 60)

    print("\n❌ 沒有 RAG（可能會幻覺）：")
    without_rag = ask_without_rag(question)
    print(without_rag)

    print("\n✅ 有 RAG（根據文件回答）：")
    with_rag = ask_with_rag(question, documents)
    print(with_rag)
    print("=" * 60)


# 測試 1：文件有的資訊

# 測試 2：文件沒有的資訊（看誰會「誠實說不知道」）
```

## 常見問題 FAQ

**Q: RAG 能完全解決幻覺問題嗎？**

不能完全解決，但能大幅降低。RAG 讓 AI 在回答時有了「參考資料」，但如果：
- 檢索到的文件不相關
- 文件本身就有錯誤
- 問題超出文件範疇

AI 還是有可能出錯。這就是為什麼第 6 章要學評估。

---

**Q: 使用 RAG 一定要用 OpenAI 嗎？**

不需要。LLM 部分可以換成任何支援聊天的模型（Claude、Gemini、Llama 等）。向量化（Embedding）的部分也可以使用本地模型（例如 sentence-transformers），這樣甚至不需要任何 API 金鑰。

---

**Q: RAG 適合用在哪些場景？**

最常見的應用：
- **企業知識庫問答**：讓員工用自然語言查詢內部文件
- **客服機器人**：根據產品說明書回答用戶問題
- **法律/醫療助手**：基於特定文件集回答專業問題
- **程式碼助手**：根據你的程式碼庫回答技術問題

---

**Q: Context Window（上下文視窗）是什麼？為什麼重要？**

Context Window 是 LLM 每次能「讀」的文字量上限。就像你的桌面空間——再大也有限。

```
GPT-4o：  128K tokens ≈ 約 10 萬中文字
Claude 3：200K tokens ≈ 約 15 萬中文字
一般 PDF：每頁約 400-600 中文字
```

即使有很大的 Context Window，把整份文件塞進去也：
- 浪費 token（成本高）
- 讓 AI 很難聚焦在相關資訊上（精確度降低）

所以我們需要 RAG 來「只選相關的部分」。

---

**Q: 第一個 Lab 的 RAG 太簡單了，這就是真正的 RAG？**

當然不是！Lab 1.1 是一個教學用的簡化版本，真正的 RAG 系統需要：
- 智慧的文件切分（第 2 章）
- 向量化和語意搜尋（第 3 章）
- 框架級別的管線管理（第 4 章）

但原理是一樣的：**查 → 加 → 答**。
