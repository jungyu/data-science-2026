# Chapter 0：AI 協作架構師宣言

> **本章定位**：價值觀校準。在學任何技術之前，先建立正確的信念系統。
>
> 「如果你只學工具，你是操作員。如果你建立原則，你是架構師。」

---

## 為什麼需要一份宣言？

想像你蓋一棟大樓。

你不會先買磚頭，你會先問：
- 這棟樓給誰住？
- 地震來了怎麼辦？
- 十年後要不要加蓋？

AI 系統也一樣。在你寫第一行 prompt 之前，你需要知道自己在蓋什麼、為什麼這樣蓋。

這份宣言就是你的「建築規範」。

---

## 五大核心信條

### 信條一：AI 是規劃器，不是決策者

```
❌ 錯誤心態：「AI 說要買這支股票，那就買吧。」
✅ 正確心態：「AI 整理了五個分析角度，讓我來判斷。」
```

**類比**：AI 就像你的研究助理。他可以幫你查資料、整理報告、跑數據。但最終簽字的人是你。

**為什麼這很重要？**

| 場景 | AI 當決策者（危險） | AI 當規劃器（安全） |
|------|---------------------|---------------------|
| 醫療診斷 | 直接告訴病人吃什麼藥 | 整理症狀、建議醫生三個可能方向 |
| 企業投資 | 自動下單買股票 | 分析報表、列出風險因子 |
| 人事決策 | 自動篩掉履歷 | 標記需要注意的資格差異 |

**實作原則**：
- 所有 AI 輸出都必須經過人類審核才能生效
- Agent 不能直接執行不可逆操作（刪除、付款、發送）
- 每個 AI 建議都要附帶「信心程度」與「替代方案」

---

### 信條二：工具優先（Tool-first）

```
❌ 錯誤做法：讓 AI 自己「想辦法」算出 KPI
✅ 正確做法：定義好 db.kpi() 工具，AI 呼叫工具取得精確數據
```

**類比**：你不會叫一個實習生「心算」公司的月營收。你會給他一台計算機、一份報表、一個 Excel 模板。

**Tool-first 的三個好處**：

```
1. 可預測性：工具的輸入輸出是固定的 schema
   → AI 不能亂回答

2. 可追溯性：每次工具呼叫都有紀錄
   → 出問題可以回溯

3. 可治理性：工具有權限控制
   → 不會越權操作
```

**反面教材**：

> 某公司讓 AI 直接寫 SQL 查詢資料庫。
> AI 寫了一個 `DELETE FROM users WHERE age > 60`。
> 因為它「理解」到題目是「清理不活躍用戶」。
> 30 萬筆資料消失了。

如果用 Tool-first：
```
tool: db.query(sql, mode="readonly")
→ AI 只能查詢，不能修改
→ 永遠不會發生上述災難
```

---

### 信條三：可追溯性 > 聰明回答

```
❌ 聰明但不可追溯：「根據我的分析，這個產品會成功。」
✅ 可追溯的回答：「根據 kb://market-report-2025.pdf (p.23)，
   該市場年增率為 12%。結合 db.kpi('customer_acquisition_cost')
   回傳的 $45，CAC payback period 為 8.2 個月。」
```

**類比**：你寫論文的時候，每個論點都要引用出處。AI 系統也一樣。

**可追溯性的三層結構**：

```
Layer 1：資料來源（Citation）
  → 這個數字從哪裡來？

Layer 2：推理過程（Reasoning Chain）
  → AI 怎麼從 A 推導到 B？

Layer 3：操作紀錄（Audit Log）
  → 誰在什麼時候做了什麼？
```

**黃金標準**：任何人看到 AI 的回答，都能在 3 分鐘內驗證其正確性。

---

### 信條四：治理 > 炫技

```
❌ 炫技：「我的 AI 可以自動寫程式、自動部署、自動回覆客戶！」
✅ 治理：「我的 AI 有三層審核、操作紀錄、回滾機制，而且每個動作都可被稽核。」
```

**類比**：銀行不會因為 ATM「很方便」就取消所有安全機制。

**治理的四個面向**：

```
┌─────────────────────────────────────────┐
│            治理四象限                      │
├──────────────┬──────────────────────────┤
│   權限控制    │    審計紀錄               │
│  誰能做什麼   │   做了什麼、什麼時候做的    │
├──────────────┼──────────────────────────┤
│   品質門檻    │    回滾機制               │
│  輸出要達標   │   出錯可以復原             │
└──────────────┴──────────────────────────┘
```

**真實案例**：

一個電商公司讓 AI 自動調整商品價格。AI 把某款商品從 $99 調到 $0.01。
三小時內被下了 50 萬筆訂單。

如果有治理機制：
- 價格變動 > 30% → 需要人工審核
- 每小時最多調整 10 個商品
- 價格變動紀錄即時推送給主管

---

### 信條五：長期可維運 > Demo 成功

```
❌ Demo 心態：「能跑就好，先給老闆看。」
✅ 維運心態：「這個系統半年後換人維護，他看得懂嗎？」
```

**類比**：你做的不是科展作品，是要上路的汽車。科展作品只要在評審面前跑一次，汽車要跑十年。

**可維運性檢查清單**：

- [ ] 新人能在 30 分鐘內理解系統架構嗎？
- [ ] 知識庫更新時，不需要改程式碼嗎？
- [ ] 工具壞了，系統會優雅降級而不是整個崩潰嗎？
- [ ] 有足夠的日誌讓你在凌晨三點除錯嗎？
- [ ] Domain Pack（領域包）可以被替換而不影響核心架構嗎？

## 宣言的實際應用

### 決策矩陣

當你面臨技術選擇時，用這個矩陣評估：

| 評估維度 | 權重 | 問自己 |
|----------|------|--------|
| 可追溯性 | 30% | 每個輸出都能追溯到來源嗎？ |
| 可治理性 | 25% | 有權限控制和審計機制嗎？ |
| 可維運性 | 25% | 半年後別人能維護嗎？ |
| 功能性   | 15% | 能解決當前需求嗎？ |
| 效率     | 5%  | 回應速度和成本可接受嗎？ |

注意：**功能性的權重只有 15%**。因為「能動」是最低標準，不是目標。

---

### 反模式清單（Anti-patterns）

| 反模式 | 描述 | 修正方向 |
|--------|------|----------|
| Prompt 萬能論 | 相信只要 prompt 寫得好就夠了 | 用工具取代 prompt hack |
| Demo 驅動開發 | 為了展示效果犧牲架構 | 先設計後實作 |
| 黑箱信仰 | 不關心 AI 怎麼得出結論 | 要求 citation 和推理鏈 |
| 全自動幻想 | 想讓 AI 全權處理所有事 | 設計 Human-in-the-loop |
| 資料樂觀主義 | 假設資料都是乾淨的 | 設計資料驗證和清理機制 |

### 練習 0-1：個人宣言草稿

用以下模板寫出你自己的三條核心信條：

```markdown
# 我的 AI 架構師宣言

## 信條一：_______
因為 _______，所以我在設計系統時會 _______。

## 信條二：_______
因為 _______，所以我在設計系統時會 _______。

## 信條三：_______
因為 _______，所以我在設計系統時會 _______。

## 我最想避免的反模式是：
_______

## 簽名：
姓名：
日期：
```

### 練習 0-2：案例分析

找一個「AI 系統出問題」的真實新聞案例（例如：自動駕駛事故、AI 面試偏見、聊天機器人洩漏隱私）。

用五大信條分析：
1. 哪條信條被違反了？
2. 如果遵守信條，結果會不同嗎？
3. 你會怎麼重新設計這個系統？
