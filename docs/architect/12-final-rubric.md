# Chapter 12：架構師導向評分標準

> **本章定位**：終點線與起跑線。讓你知道怎麼被評量，以及怎麼超越期待。
>
> 「知道終點在哪裡的人，跑起來特別有方向感。」

---

## 評分哲學

### 這不是考試，是專業審查

```
傳統考試：
  「答對幾題？」
  → 測量你記住了多少

架構師評審：
  「你的系統設計考慮了什麼？」
  → 測量你思考了多少

差異：
  考試可以靠背誦得高分
  架構審查只能靠理解和判斷
```

### 評分的核心理念

```
我們不是在問：「你的系統能不能跑？」
我們是在問：「你的系統值不值得讓別人繼續維護？」

能跑 = 最低標準
能維護 = 及格
能演進 = 優秀
能治理 = 頂尖
```

## 維度一：架構完整度（25%）

### 核心問題
> 你的系統設計是否完整且合理？每一層的責任是否清楚？

### 評分細則

```
A+（95-100）— 超越期待
┌──────────────────────────────────────────────────┐
│ ✓ 完整的四層架構圖（User → Orchestrator →        │
│   Services → Data），每層職責明確                  │
│ ✓ 三大支柱（RAG + MCP + Automation）設計完整     │
│ ✓ 故障隔離設計：Circuit Breaker + 降級策略        │
│ ✓ 資料流圖展示完整的請求旅程                      │
│ ✓ 有 ADR 記錄關鍵架構決策                        │
│ ✓ 展示了五年演進的考量                            │
└──────────────────────────────────────────────────┘

A（85-94）— 優秀
┌──────────────────────────────────────────────────┐
│ ✓ 完整的架構圖，每層職責清楚                      │
│ ✓ 三大支柱有基本設計                              │
│ ✓ 有故障處理策略                                  │
│ ✓ 有資料流圖                                     │
└──────────────────────────────────────────────────┘

B（75-84）— 良好
┌──────────────────────────────────────────────────┐

│ ✓ 主要組件有設計                                  │

┌──────────────────────────────────────────────────┐

│ ✗ 但缺乏層次分離                                 │

## 維度二：工具契約設計（15%）

### 核心問題
> 你的工具設計是否標準化、安全、且好用？

### 評分細則

```
A+：
  ✓ 至少 5 個工具有完整的六要素契約
  ✓ 命名遵循 domain.action 慣例
  ✓ Input 使用 JSON Schema + enum + pattern
  ✓ Output 結構化且包含 source
  ✓ 權限和風險分級明確
  ✓ 錯誤處理有分類、有建議
  ✓ 有版本號管理

A：
  ✓ 至少 3 個工具有完整契約
  ✓ 命名一致
  ✓ 有 schema 定義
  ✓ 有權限分級

B：
  ✓ 工具有基本定義
  ✓ 有一些 schema
  ✗ 缺少權限或風險分級

C：
  ✓ 有工具概念
  ✗ 沒有標準化的契約
  ✗ 缺少安全設計
```

  ✓ 五道 anti-hallucination 防線
  ✓ 知識庫有版本控制
  ✓ Chunk 策略有設計理由

A：
  ✓ Citation 機制完整
  ✓ 基本的 anti-hallucination 設計
  ✓ 知識庫有版本管理

B：
  ✓ 有 citation 但格式不一致
  ✓ 基本的 RAG 流程
  ✗ 缺少 hallucination 防護

C：
  ✓ 有 RAG 概念
  ✗ 沒有 citation
  ✗ 沒有 hallucination 防護
```

## 常見扣分陷阱

### 十大常見問題

```
1.「只有 demo，沒有架構」
   → 系統能跑不代表架構好。需要四層圖 + 責任邊界。

2.「工具沒有 schema」
   → 每個工具必須有標準化的 input/output 定義。

3.「回答沒有 citation」
   → 這是 A2RM 的核心要求，不能省。

4.「沒有權限控制」
   → AI 預設必須是 readonly。

5.「沒有審計紀錄設計」
   → 出事了怎麼追溯？

6.「自動化流程沒有異常處理」
   → Happy path 容易，error handling 才見功夫。

7.「沒有成本意識」
   → Token 不是免費的。

8.「ADR 只寫了結論」
   → 要寫「為什麼」而不只是「做了什麼」。

9.「Domain Pack 不完整」
   → 9 個文件缺一不可。

10.「文件給自己看的，別人看不懂」
   → 文件是寫給六個月後的陌生人看的。
```

## 從 A 到 A+ 的秘訣

### 超越期待的方式

```
1. 展示思考深度
   不只說「我選了 X」，說「我考慮了 A, B, C，
   因為 [權衡分析]，所以選了 X，代價是 [後果]」

2. 展示演進思維
   「這個設計在 1000 用戶時可以這樣，
    10000 用戶時需要這樣調整」

3. 展示治理意識
   「這個功能如果出錯，影響範圍是 ___，
    所以我設計了 ___ 來防護」

4. 展示真實場景
   「我模擬了 ___ 攻擊場景，
    我的 guardrails 成功擋下了 ___」

5. 展示反思能力
   「如果重做一次，我會 ___ 不同，
    因為我發現 ___」
```

## 面試武器

### 完成這個課程後，你可以在面試時說：

```
「我設計過一個完整的 AI × RAG × MCP 系統。」

「我為系統建立了工具契約標準，
 每個工具都有 schema、權限分級和風險評估。」

「我設計了三層 Guardrails 來防止 hallucination 和資料洩漏。」

「我寫過 ADR 來記錄架構決策的權衡分析。」

「我理解 AI 系統的成本結構，
 並設計了 Token 追蹤和預算控制機制。」

「我知道什麼是 Data Drift，
 以及怎麼偵測和處理系統的漸進式退化。」
```

這些，比「我會用 LangChain」強太多了。
